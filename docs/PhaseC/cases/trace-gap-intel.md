## 2. Top claim (G1)
The trace_gap_intel v0.1.0 is safe to use as an **advisory ranker** for
which requirements may be impacted by a code diff. It never auto-merges,
auto-waives, or auto-approves any requirement, test, or change request.

## 3. Context
- **C1** Operational environment: code review pipeline. Output is a
  ranked list shown in the PR template.
- **C2** Upstream: requirement IDs from `docs/M1/requirements/requirements.csv`
  + the file paths in the PR diff.
- **C3** Downstream: human reviewer chooses whether to act on the
  suggestion.
- **C4** DAL allocation: **N/A**.
- **C5** Data sources: requirement CSV + curated synthetic diff cases
  used to compute precision_at_3 and MRR.
- **C6** User population: PR reviewers + verification engineers.

## 4. Assumptions (falsifiable)
- **A1** The module-prefix → requirement-prefix table reflects current
  module organisation. (Currently a small static map.)
- **A2** Reviewers always read the requirements list themselves; the
  ranker does not relieve them of responsibility.
- **A3** No automated downstream tool calls `rank_impacted_requirements`
  in a way that would make the ranking authoritative.

## 5. Strategy (S1)
G1 decomposes into:
- G1.1 — output is a ranked list, never a binary decision
- G1.2 — baseline accuracy is measured on a known small set
- G1.3 — interface accepts a learned ranker without changing call sites
- G1.4 — model never gates a merge

## 7. Failure modes
- **F1** Module map drift: file moves silently invalidate the prefix map.
  Mitigation: linter that checks every module-prefix path actually exists.
- **F2** New requirement prefix added but not mapped → silent under-recall.
  Mitigation: case re-review whenever `requirements.csv` adds a new
  prefix family.
- **F3** Reviewers over-trust the top-1 suggestion → bias.
  Mitigation: UI presents top-K with the DRAFT banner.

## 8. Fallback behavior
If the ranker is unavailable, reviewers fall back to manual change-impact
analysis using the `docs/M1/certification/change-procedure.md` checklist —
the procedure that existed before Phase B.

## 9. Human override path

- Acceptance/rejection of suggested requirements is recorded in the PR
  conversation; explicit overrides flow into the `human_review` Silver
  table when promoted.
